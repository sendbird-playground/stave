# Lean Cross-Provider Task Context Plan

## Objective

Reduce token waste and context noise when switching a single Stave task between Claude and Codex.

The immediate goal is not full agent-to-agent collaboration. The goal is a small, deterministic, Stave-owned context layer that lets the next provider continue useful work without replaying a long raw transcript.

## Current Problem

Today cross-provider continuation leans too heavily on rendered prompt text and transcript replay.

That creates three issues:

1. Too many tokens are spent on raw history instead of the current task state.
2. Provider switches inherit noisy chat rather than a compact working summary.
3. The implementation risk grows quickly if Stave adds multiple new memory layers too early.

## Constraints

- Keep SQLite as the durable source of truth for Stave state.
- Keep provider-native session ids provider-local.
- Do not require Claude or Codex to read SQLite directly.
- Avoid introducing semantic retrieval, MCP write paths, or A2A-style collaboration in v1.
- Reuse existing Stave structures where possible: task thread, turn persistence, `request_snapshot`, and provider-specific resume handling.

## Non-Goals

- Sharing a Claude native session with Codex or vice versa.
- Transferring hidden reasoning across providers.
- Building a multi-layer memory system in the first iteration.
- Adding a new vector store or analytics database for this feature.

## Lean Proposal

### Single Durable Snapshot

Add one task-scoped durable object: `task_context_snapshot`.

Suggested fields:

- `objective`
- `current_state`
- `facts`
- `decisions`
- `open_questions`
- `artifacts`
- `next_action`
- `updated_at`
- `source_turn_id`

This snapshot becomes the compact Stave-owned cross-provider handoff state.

### No Separate Handoff Table In v1

Do not add a dedicated handoff packet store in the first phase.

Instead:

- keep `task_context_snapshot` as the single cross-provider state object
- reuse existing turn persistence and `request_snapshot` for audit history
- derive any human-readable handoff text from the snapshot when needed

### Provider Switch Injection

When the provider changes inside the same task:

1. continue to use only the selected provider's own native session id
2. inject current task awareness as today
3. inject `task_context_snapshot`
4. keep transcript/history replay bounded and secondary

This preserves current provider-local resume behavior while removing pressure to replay large raw history across vendors.

## Data Model

Store the snapshot in SQLite alongside existing task/workspace persistence.

Minimum additions:

- `taskContextSnapshotByTask` in runtime state or workspace projection
- a persisted task-scoped snapshot record in SQLite
- schema validation in the existing task-context schema layer

No new retrieval index, artifact index, or external memory store is required for the first iteration.

## Update Strategy

Use deterministic update logic on completed turns.

Rules:

- update only on completed assistant turns
- derive fields from visible turn output, persisted artifacts, and explicit task state
- prefer dropping stale detail over keeping an oversized snapshot
- cap field sizes so prompt assembly remains stable

## Prompt Assembly Changes

Refine the provider request pipeline so cross-provider continuation prefers compact state over raw replay.

Target order:

1. current task awareness
2. `task_context_snapshot`
3. bounded referenced-task context if explicitly requested
4. current user input
5. bounded recent history only when still useful

This should replace the implicit assumption that large history replay is the default continuation path.

## Rollout Plan

### Phase 0: Instrumentation

Measure the current baseline before changing behavior.

Track at minimum:

- injected context size per turn
- history replay size per provider switch
- success rate of first cross-provider continuation turn
- cases where a provider switch still needs manual restatement

### Phase 1: Snapshot Persistence

- add `task_context_snapshot` schema
- persist it in SQLite
- update it on completed turns
- keep the feature internal and non-editable

### Phase 2: Provider Switch Injection

- inject the snapshot during cross-provider turns
- keep provider-local native session resume unchanged
- reduce replayed history when the snapshot is present

### Phase 3: Read-Only Inspection

- surface the snapshot in a minimal inspectable UI or debug view
- do not allow direct provider writes yet
- evaluate whether a small `latest_turn_delta` is still necessary

## Success Criteria

- median injected context size during provider switch is materially lower than the raw replay baseline
- Claude and Codex can continue the same task with less manual restatement
- the implementation stays within the existing SQLite-backed architecture
- no extra persistence systems are introduced for v1

## Risks

- snapshot drift if update rules are too loose
- over-compression that drops an important recent decision
- pressure to treat the snapshot as user-editable too early

## Deferred Work

Only revisit these after the lean snapshot approach is measured in practice:

- dedicated handoff delta records
- MCP read/write access for task memory
- semantic retrieval for old turns or artifacts
- background consolidation workers
- A2A-style provider collaboration primitives
